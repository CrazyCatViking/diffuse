Diffuse Research Note

Better Diffs For Code Review

Diffuse currently renders Git unified diffs. That is reliable and patch-correct, but it does not explain which tokens changed, which blocks moved, or how related edits fit together. The goal should be a layered diff model: Git for correctness, token spans for precision, and syntax-aware relationships for understanding.

Keep Git Use Git as the canonical patch and file-selection layer.
Add Detail Layer line pairing, token spans, moved blocks, and confidence on top.
Use Trees Use tree-sitter selectively for structural grouping and semantic explanations.

Current Diffuse Pipeline

The current architecture is intentionally simple: ask Git for a unified diff, parse the patch text, and render rows. This is dependable, but the resulting model only knows about hunk rows, context rows, additions, and deletions.

1

Git Diff

git diff or git diff -U999999 produces patch text for a target file.

2

Unified Parser

core/src/core/diff.zig parses hunk headers and line prefixes.

3

Render Model

Rows carry kind, line numbers, line text, hunk headers, and syntax spans.

4

Viewer

The app renders split or inline rows and lazily overlays syntax highlighting.

Important limitation: this model cannot represent paired replacements, token-level changes, moved code, copied code, format-only edits, or relationships across non-adjacent hunks.

Existing Alternatives

There is no single perfect engine. The useful products split into three groups: Git-style text diff, syntax-aware display diff, and AST edit-action diff.

Option Strengths Weaknesses Best Use
Git Diff Fast, installed everywhere, patch-correct. Supports --histogram, --patience, --word-diff, --color-moved, -M, and -C. Still line-oriented. Word and moved-line output are awkward as durable structured data. Canonical patch layer, baseline display, and fallback.
libgit2 / xdiff Programmatic access to Git-style file, hunk, and line diffs. Mostly the same line-oriented model. Adds native dependency complexity. Potential future replacement for shelling out to Git, not a semantic diff solution.
Difftastic Excellent human structural diffs using tree-sitter. Handles wrapping and formatting better than line diff. CLI and display oriented, not patch-compatible, no stable structured API, performance concerns on large changes. Benchmark, inspiration, and possible optional experimental engine.
GumTree AST edit actions such as insert, delete, update, move, and rename. Strong research foundation. Heavy integration, Java runtime, LGPL licensing considerations, uneven language ergonomics. Reference model for move detection and AST action vocabulary.
Monaco / VS Code Diff Mature text diff UX with good intraline matching. Not AST-aware and would fight Diffuse's custom renderer. UX inspiration, not a core replacement.
CodeMirror Merge Good editor-side merge and diff primitives. Text-focused and expensive to integrate if Diffuse keeps its current viewer. Useful only if moving toward a CodeMirror-based diff surface.
Token Diff Simple, fast, and precise enough to answer which token was deleted or replaced. Not semantic. Cannot reliably understand moved functions or syntax wrappers. Best first step for high-value intraline clarity.
Tree-sitter Custom Diff Fits Diffuse's existing syntax infrastructure. Can group edits by symbol and syntax range. Hard to produce ideal output in every case. Needs careful fallbacks and confidence scoring. Best long-term path for Diffuse-owned code understanding.

Recommended Architecture

Build a Diffuse-owned diff intelligence layer. Keep Git as the source of truth and add richer analysis as annotations rather than replacing the patch pipeline all at once.

Patch Layer

Correctness and fallback

Git decides file selection, old/new content boundaries, hunks, line numbers, staged and unstaged behavior, renames, and applyable patch semantics.

Token Layer

Precise local changes

Pair nearby deleted and added lines, tokenize both sides, and produce deleted, inserted, and replaced spans that the existing CodeText highlight pipeline can render.

Relationship Layer

Why changes belong together

Detect moved blocks, copied blocks, changed symbols, identifier renames, format-only groups, and syntax-level wrappers using tree-sitter when confidence is high.

Principle Do not parse pretty output

Git color output and Difftastic terminal output are useful for experiments, but Diffuse needs a stable structured JSON model that the UI can reason about.

Principle Prefer confidence over cleverness

When semantic analysis is uncertain, show the reliable Git diff plus token highlights. A wrong semantic explanation is worse than no semantic explanation.

Richer Data Model

The important design move is to separate rendered rows from relationships. Rows remain stable for review anchors and scrolling, while annotations connect ranges across rows, hunks, and files.

tokenSpans

Column ranges inside old or new text. Kinds include inserted, deleted, replaced, whitespace-only, and unchanged-inside-edit.

linePairs

Explicit old/new line relationships. This distinguishes a replacement from unrelated delete and add rows.

changeGroups

Group related rows and spans under one id. Examples: moved block, rename, extracted helper, formatter-only change.

relations

Cross-range links such as moved-from, moved-to, copied-from, renamed-from, wraps, unwraps, and same-symbol.

symbols

Containing function, class, method, import block, object key, or test block derived from tree-sitter.

confidence

Every non-Git explanation should carry confidence and engine metadata so the UI can choose how strongly to present it.

Implementation Roadmap

The safest path is incremental. Each stage should improve the product while preserving Git fallback behavior and review anchor stability.

Phase 1: Better Git Baseline

Experiment with --diff-algorithm=histogram for display diffs, keep current parsing, and build a benchmark corpus of tricky examples.

Phase 2: Token-Level Highlighting

Pair nearby deleted and added lines, run a token diff, add diff highlight kinds to CodeText, and show exact inserted or deleted tokens.

Phase 3: Moved-Block Detection

Normalize changed lines, hash deleted and added blocks, detect same-file moves first, then expand to moved-and-edited and cross-file cases.

Phase 4: Syntax-Aware Grouping

Use tree-sitter to label changes by containing symbol and detect high-confidence wrapper, rename, import reorder, and format-only patterns.

Risks And Constraints

Rich diffs can become misleading if the engine over-explains. The implementation should be conservative, measurable, and easy to fall back from.

Column Coordinates

Tree-sitter byte columns, Zig slices, JavaScript UTF-16 indexes, and Unicode text can diverge. Token spans need one documented coordinate system.

Performance

Structural diffing can be expensive on large files and large rewrites. Add file-size, hunk-size, and parse-error fallbacks from day one.

False Relationships

Move detection can connect unrelated repeated code. Require confidence thresholds and make uncertain relationships visually subtle.

Review Anchors

Existing anchors depend on side, line, selected text, and diff target fingerprint. Enhanced annotations must not destabilize comments.

Practical Next Steps

The first useful milestone is not a full AST diff engine. It is a richer model with precise token highlights and a benchmark set that lets us evaluate improvements objectively.

Create Benchmarks

Add fixtures for formatter-only changes, moved blocks, moved-and-edited code, identifier rename, wrapper added, import reorder, and large rewrites.

Add Token Spans

Extend the render model with optional token annotations and add a diff highlight kind to the existing code text renderer.

Measure Engines

Compare current Git, Git histogram, Difftastic, GumTree, and the Diffuse token engine against the benchmark corpus.

End state: Diffuse should show the exact patch, the exact token changes, and the relationships that make a review easier to reason about.
